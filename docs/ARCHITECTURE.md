# Architecture: Shared State (Blackboard) Pattern for Multi-Agent PR Review

## Overview

This document explains the architectural design of the agentic-review-gate system, focusing on why we chose the **Shared State (Blackboard) Pattern** and how it solves the "message pass-through fatigue" problem.

## Problem Statement

### Traditional Linear Agent Chains

Naive multi-agent systems often chain agents sequentially:

```
Agent A вҶ’ Agent B вҶ’ Agent C
  вҶ“        вҶ“        вҶ“
Input   Transform  Output
```

**Problems:**
1. **Context Loss**: Agent C only sees Agent B's output, not Agent A's original analysis
2. **Pass-Through Fatigue**: Information degrades through multiple transformations
3. **Tight Coupling**: Each agent must understand the output format of the previous agent
4. **Sequential Execution**: No parallelism; total time = sum of all agent times
5. **Limited Synthesis**: Final agent can't reason about original findings

**Example:**
```
Logic Agent finds: "Method has 10 branches"
  вҶ“ (simplified to summary)
Security Agent sees: "Complex code detected"
  вҶ“ (doesn't know WHY it's complex)
Summarizer sees: "Complex code detected"
  вҶ“ (can't make intelligent deductions)
Final comment: "Code is complex" (vague, not actionable)
```

## The Blackboard Pattern Solution

### Architecture

Instead of a chain, we use a **shared ReviewState** ("Blackboard") where all agents deposit findings:

### Key Characteristics

1. **Immutable Input**: `pr_metadata` is read-only
2. **Append-Only Findings**: Agents add findings; they never remove or modify others' findings
3. **Execution Metadata**: Each agent records execution time, status, token usage
4. **Control Flags**: `is_blocked` flag set when critical issues detected
5. **Final Synthesis**: `final_summary` generated by Summarizer from all findings

## Workflow Phases

### Phase A: Parallel Analysis

Both agents run concurrently on the same state snapshot:

```python
async def _phase_a_analyze(self, state: ReviewState) -> None:
    # Run both agents in parallel
    results = await asyncio.gather(
        logic_agent.execute(state),
        security_agent.execute(state)
    )
    
    # Merge results back
    for findings, metadata in results:
        state.add_finding(findings)
        state.add_metadata(metadata)
```

**Time Complexity:**
- Sequential: Time(Logic) + Time(Security) + Time(Summarizer)
- Parallel: max(Time(Logic), Time(Security)) + Time(Summarizer) вүҲ **50% faster**

**Benefits:**
- Both agents see identical context
- No information degradation
- Neither agent is blocked by the other
- Flexible to add more agents (Performance = max agent time, not sum)

### Phase B: Critical Issue Evaluation

Check if any findings warrant blocking the PR:

```python
async def _phase_b_evaluate(self, state: ReviewState) -> None:
    critical_findings = state.get_critical_findings()
    if critical_findings:
        state.set_blocked(True)
```

This flag influences the tone of the final summary.

### Phase C: Synthesis

The Summarizer reads ALL accumulated findings and produces a single, cohesive comment:

```python
async def _phase_c_synthesize(self, state: ReviewState) -> None:
    comment = await summarizer.generate_comment(state)
    state.set_summary(comment)
```

**Summarizer Responsibilities:**

1. **Deduplication**: Remove duplicate findings
   ```python
   def _deduplicate_findings(self, findings: List[AgentFinding]):
       # If Logic and Security both flag line 42, keep one copy
       # with a note that multiple agents detected it
   ```

2. **Organization**: Group by severity and file
   ```
   Critical Issues
   в”ңв”Җ file1.py
   в”Ӯ  в””в”Җ SQL Injection (line 15)
   в””в”Җ file2.py
      в””в”Җ Hardcoded Secret (line 8)
   
   Warnings
   в”ңв”Җ file3.py
   в”Ӯ  в””в”Җ High Complexity (line 42)
   ```

3. **Professional Tone**: Convert technical findings to peer-level communication
   ```
   вқҢ Bad: "Cyclomatic complexity exceeds threshold"
   вң… Good: "Consider extracting conditional logic into helper methods"
   ```

4. **Ranking**: Critical issues first
   ```
   1. [CRITICAL] Security issues (must fix before merge)
   2. [WARNING] Quality issues (should address)
   3. [INFO] Suggestions for improvement
   ```

## Data Model

### ReviewState (The Blackboard)

```python
class ReviewState(BaseModel):
    # Input (immutable after creation)
    pr_metadata: PRMetadata
    
    # Mutable findings (append-only)
    findings: List[AgentFinding] = []
    
    # Execution records
    metadata: List[AgentMetadata] = []
    
    # Control flags
    is_blocked: bool = False
    
    # Output
    final_summary: Optional[str] = None
    
    # Tracking
    created_at: datetime
    updated_at: datetime
```

### AgentFinding

The fundamental unit of analysis:

```python
class AgentFinding(BaseModel):
    file_path: str              # What file
    line_number: Optional[int]  # Where (optional)
    finding_type: str           # Category of finding
    description: str            # What is the issue
    suggestion: str             # How to fix it
    severity: Severity          # CRITICAL | WARNING | INFO
    agent_id: str               # Which agent found this
```

### Severity Levels

- **CRITICAL**: Blocks PR merge; security or structural issues
- **WARNING**: Should be addressed; code quality or maintainability
- **INFO**: Suggestions for improvement; nice-to-haves

## Agent Interface

All agents inherit from `BaseAgent`:

```python
class BaseAgent(ABC):
    async def analyze(self, state: ReviewState) -> List[AgentFinding]:
        """
        Analyze PR and return findings.
        
        - Receives a snapshot of current state
        - Does NOT modify state directly
        - Returns findings for coordinator to merge
        """
        pass
    
    async def execute(self, state: ReviewState) -> Tuple[List[AgentFinding], AgentMetadata]:
        """
        Wrapper that calls analyze() and records metadata.
        """
        findings = await self.analyze(state)
        metadata = AgentMetadata(...)
        return findings, metadata
```

### Agent Responsibilities

Each agent is **responsible for one domain**:

1. **Logic Agent**
   - Design patterns
   - SOLID violations
   - Code complexity
   - Dead code
   - Architecture concerns

2. **Security Guard Agent**
   - Hardcoded secrets
   - PII leaks
   - OWASP vulnerabilities
   - Insecure patterns

3. **Summarizer Agent** (different pattern)
   - Does NOT generate new findings
   - Synthesizes findings into output
   - Intelligent deduplication
   - Professional tone

## Extensibility

### Adding a New Agent

1. Create `src/code_reviewer/agents/custom.py`:
```python
class CustomAgent(BaseAgent):
    def __init__(self):
        super().__init__(agent_id="custom")
    
    async def analyze(self, state: ReviewState) -> List[AgentFinding]:
        findings = []
        for file in state.pr_metadata.files_changed:
            # Your analysis here
            if issue_detected:
                findings.append(
                    self._create_finding(
                        file_path=file,
                        finding_type="...",
                        description="...",
                        suggestion="...",
                        severity=Severity.WARNING,
                    )
                )
        return findings
```

2. Register in `ReviewCoordinator`:
```python
class ReviewCoordinator:
    def __init__(self):
        self.custom_agent = CustomAgent()
        # ... other agents ...
    
    async def _phase_a_analyze(self, state: ReviewState):
        agents = [
            self.logic_agent,
            self.security_agent,
            self.custom_agent,  # Add here
        ]
        # Execute all in parallel
```

### Integrating LLM Analysis

Agents can use LLMs by overriding `analyze()`:

```python
class LLMAgent(BaseAgent):
    async def analyze(self, state: ReviewState) -> List[AgentFinding]:
        # Use Claude, GPT-4, etc.
        response = await self.llm.analyze(
            files=state.pr_metadata.files_changed,
            diff=state.pr_metadata.diff_content,
        )
        
        # Parse response into AgentFinding objects
        findings = self._parse_response(response)
        return findings
    
    def _get_token_usage(self) -> dict[str, int]:
        return {
            "input_tokens": self.last_input_tokens,
            "output_tokens": self.last_output_tokens,
        }
```

Token usage is automatically tracked in `AgentMetadata`.

## Error Handling

Each agent's execution is wrapped with error handling:

```python
async def execute(self, state: ReviewState) -> Tuple[List, AgentMetadata]:
    try:
        findings = await self.analyze(state)
        status = "success"
    except Exception as e:
        findings = []
        status = "error"
        error_message = str(e)
    
    metadata = AgentMetadata(
        agent_id=self.agent_id,
        status=status,
        error_message=error_message,
        ...
    )
    return findings, metadata
```

If an agent fails, the review continues with its findings marked as empty, but the error is recorded.

## Performance Characteristics

### Timing

With heuristic-based agents (no LLM):
- Logic Agent: ~200-500ms
- Security Guard: ~200-500ms
- Summarizer: ~50-100ms
- **Total (Sequential)**: ~450-1100ms
- **Total (Parallel)**: ~250-600ms вң… ~50% faster

With LLM-based agents (Claude, GPT-4):
- Logic Agent: ~3-5s
- Security Guard: ~2-4s
- Summarizer: ~100-200ms
- **Total (Sequential)**: ~5-10s
- **Total (Parallel)**: ~3-5s вң… ~50-60% faster

### Memory

- ReviewState: O(f) where f = number of findings
- Per agent execution: O(1) - stateless
- Total: Typically < 10MB for typical PRs

### When to Extract Into New Module

Move code when:
- вң… It has a single, clear responsibility
- вң… It could be tested independently
- вң… It might be reused elsewhere
- вң… It has multiple classes/functions (>300 LOC)

Don't extract prematurely:
- вқҢ Avoid micro-modules for single functions
- вқҢ Don't split code that's always used together
- вқҢ Wait until you see the pattern (rule of three)

## Design Principles

### Dependency Inversion

High-level modules should NOT depend on low-level implementation details.

**Bad:**
```python
class LogicAgent:
    def __init__(self):
        self.github = GitHubClient()  # Concrete dependency
        self.llm = get_llm_client()    # Function dependency
```

**Good:**
```python
from core.interfaces import GitHubAPIClient, LLMClient

class LogicAgent:
    def __init__(self, github: GitHubAPIClient, llm: LLMClient):
        self.github = github
        self.llm = llm
```

### Strategy Pattern for Agents

Agents should use consistent interface for different analysis strategies.

**All agents must implement:**
```python
from typing import List
from core.state import ReviewState
from base import AgentFinding

class AnalysisAgent:
    async def analyze(self, state: ReviewState) -> List[AgentFinding]:
        """Analyze PR and return findings."""
        ...
```

### Observer/Event Pattern for Webhooks

Webhook handling should be decoupled from review execution.

**Current pattern:**
```
WebhookHandler вҶ’ ReviewRequest вҶ’ BackgroundTask вҶ’ ReviewCoordinator
```

Maintains separation of concerns:
- WebhookHandler: Validates and parses GitHub events
- ReviewRequest: Data transfer object
- BackgroundTask: Async execution management  
- ReviewCoordinator: Orchestrates agent analysis

## Contributing Guidelines

### Pull Request Guidelines

#### PR Size & Scope

**Follow the Single Responsibility Principle at the PR level:**

Each PR should address ONE concern. Avoid large PRs that mix multiple independent changes.

**Recommended PR breakdown:**

1. **Infrastructure PRs** (Testing, Configuration)
   - Files: `tests/`, `TESTING.md`, `pytest.ini`, `.github/workflows/`
   - Example title: `test: Add webhook integration tests`

2. **Core Feature PRs** (Agent Logic, Coordinator)
   - Files: `agents/`, `core/coordinator.py`, `core/interfaces.py`
   - Example title: `feat: Implement security analysis agent with pattern matching`

3. **Utility PRs** (GitHub Client, Diff Parser, Cache)
   - Files: `utils/`, `core/interfaces.py`
   - Example title: `fix: Improve GitHub API diff retrieval with proper headers`

4. **Architecture PRs** (Interfaces, Patterns, Refactoring)
   - Files: `core/interfaces.py`, dependency injection changes
   - Example title: `arch: Add dependency inversion interfaces for testability`

**Benefits:**
- вң… Easier to review thoroughly
- вң… Reduces risk of introducing bugs
- вң… Clearer git history for bisecting issues
- вң… Simpler to revert if needed
- вң… Enables parallel review and merging

#### PR Title Format

Use the format: `[type]: [scope] - [description]`

Examples:
```
fix: GitHub API diff retrieval - handle empty responses gracefully
feat: Logic agent - add SOLID principle violation detection
test: Webhook integration - add PR comment validation tests
arch: Dependency inversion - introduce abstraction interfaces
docs: Contributing guide - clarify PR structure expectations
refactor: Agent base - extract common logging patterns
perf: Diff parser - optimize hunk iteration performance
```

**Types:**
- `feat`: New feature
- `fix`: Bug fix
- `test`: Test additions/improvements
- `arch`: Architectural changes
- `refactor`: Code restructuring
- `perf`: Performance improvements
- `docs`: Documentation
- `ci`: CI/CD configuration

### Code Review Checklist

When reviewing PRs, verify:

- [ ] Single Responsibility Principle: Does PR address one concern?
- [ ] Architecture: Are high-level modules free of low-level dependencies?
- [ ] Testing: Are critical changes covered by tests?
- [ ] Documentation: Is PR title clear and specific?
- [ ] Logging: Are important operations logged for debugging?
- [ ] Error Handling: Are error cases handled gracefully?

### Commit Message Guidelines

Write clear, descriptive commit messages:

```
[type]: [scope] - [description]

[Extended explanation if needed]

Related issues: #123, #456
```

Examples:
```
fix: GitHub client - use correct Accept header for diff retrieval

The v3.diff format was deprecated. Updated to use modern GitHub API
header format (application/vnd.github.diff) which returns diffs 
consistently across API versions.

Fixes: Diff content showing as empty in PR reviews
```

### Testing Requirements

- Unit Tests: Test individual agents with mocked dependencies
- Integration Tests: End-to-end webhook processing
- Webhook Tests: Manual testing with ngrok tunnel

See [OPERATIONS.md#testing](OPERATIONS.md#testing) for detailed procedures.

### Performance Considerations

#### Agent Execution Time
- Target: < 30 seconds per PR analysis
- Profile with: `time_execution()` decorator in logger
- Optimize before: > 60 seconds per PR

#### GitHub API Rate Limits  
- Get PR info: 1 call
- Get PR diff: 1 call
- Post comment: 1 call
- Total: 3 calls per PR (safe within 5000/hour limit)

#### Memory Usage
- Diff content: Usually < 1MB
- Cache: In-memory by default (suitable for small deployments)
- Consider Redis for production (> 100 PRs/day)

## Security Considerations

1. **Input Validation**: Validate PR diffs for malicious content
2. **Secret Scanning**: Don't log or expose secrets
3. **Rate Limiting**: Prevent abuse
4. **Authentication**: Validate GitHub webhooks
5. **Output Sanitization**: Escape user content in comments
